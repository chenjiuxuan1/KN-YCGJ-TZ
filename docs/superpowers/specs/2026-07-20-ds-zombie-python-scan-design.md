# DS 僵尸任务 Python 扫描改造设计

## 1. 背景与目标

现有 n8n 工作流通过 SSH 导出各国 DolphinScheduler 元数据，将完整明细序列化到 stdout，再由 n8n JavaScript 节点解析、聚合并传给评分子流程。单国结果已达到约 20.7 MB，造成浏览器卡顿、节点内存压力和 502。

本次采用方案 A：各国跳板机上的 Python 程序直接查询 DS 元数据库，在本地完成工作流聚合、上下游解析、SOP 评分和治理明细落库；n8n 仅负责触发、传参和接收小体积摘要。

目标：

- 大体量 DS 元数据不进入 n8n 节点链路。
- 以工作流为治理对象，同时保留任务级证据。
- 同时识别任务 DAG 和跨工作流上下游。
- 根据 SOP 输出可解释的 A/B/C/D 等级与建议动作。
- 任何下线动作都只生成候选和操作计划，不自动修改 DS。
- 数据库密码等敏感配置仅从环境变量读取，不进入仓库或 n8n workflow JSON。

## 2. 范围与边界

### 2.1 本期范围

- 支持中国、菲律宾、印尼、墨西哥、泰国和巴基斯坦。
- 只读查询 DS 项目、流程定义、任务定义、调度、实例和依赖元数据。
- 计算工作流活跃度、失败情况、调度状态和上下游关系。
- 将批次、候选、任务证据和依赖证据写入治理库。
- 生成 n8n 可安全接收的汇总 JSON 和 Top 20 预览。
- 改造现有手动触发 n8n workflow，使其调用 Python 扫描入口。

### 2.2 不在本期自动执行的事项

- 不自动停调度、下线、删除或归档工作流。
- 不把“未查到访问记录”等同于“无人使用”；缺少外部证据时记录为 `unknown`。
- 不在代码或 workflow 中保存数据库密码。
- 不跨国集中拉取全部原始任务 JSON。

## 3. 总体架构

```text
n8n 手动/定时触发
  -> 按国家路由到 SSH 节点
  -> 跳板机调用 ds_zombie_scan.py
  -> Python 分批读取 DS 元数据库
  -> 聚合工作流和任务证据
  -> 解析任务级及跨工作流依赖
  -> 计算等级、保护状态和建议动作
  -> 事务写入治理库
  -> stdout 仅输出汇总 JSON
  -> n8n 展示结果或进入后续通知流程
```

扫描使用批次 ID 保证可追踪与幂等。相同 `country + batch_id + workflow_code + score_version` 重跑时更新本批结果，不产生重复候选。

## 4. Python 组件

### 4.1 扫描入口

新增 `remote_scripts/ds_zombie_scan.py`，参数包括：

- `--country`
- `--batch-id`
- `--min-stale-months`，默认 3
- `--lookback-days`，默认 30
- `--project-name`、`--workflow-name`、`--task-name`
- `--score-version`，默认 `v1`
- `--write-to-db`，显式开启才写治理库
- `--top-limit`，默认 20，最大 100
- `--dry-run`，完成查询与评分但不写治理库

DS 连接从 `DS_DB_HOST`、`DS_DB_PORT`、`DS_DB_DATABASE`、`DS_DB_USER`、`DS_DB_PASSWORD` 读取。治理库连接使用独立的 `GOVERNANCE_DB_*` 环境变量。

### 4.2 数据访问层

新增可测试的数据访问层，负责：

- 识别当前国家使用的 DS 表结构版本与可用字段。
- 使用服务端分页或游标分批读取，禁止一次性加载全库原始行。
- 查询项目、工作流、任务、调度、流程实例、任务实例和依赖关系。
- 对缺表或版本差异返回结构化能力状态，不静默降级。

所有 SQL 使用参数绑定；用户输入只作为过滤值，不拼接为 SQL 结构。

### 4.3 工作流聚合器

以 `country + project_code + workflow_code` 为唯一对象，聚合：

- 项目、工作流、任务名称和负责人。
- 工作流、调度上线状态。
- 创建、更新、最近运行、最近成功和最近失败时间。
- 近 30 天总运行、成功、失败次数。
- 任务数量、任务类型和资源引用摘要。
- 外部异常 SQL、查询访问和 Superset 访问证据的状态。

外部证据使用三态：`present`、`absent`、`unknown`。只有完成相应数据源扫描后才能标为 `absent`。

## 5. 上下游识别

上下游是治理保护证据，必须在评分前完成。

### 5.1 任务级 DAG

解析流程定义中的任务关系表或等价关系数据，建立：

- 工作流内部任务直接上游和下游。
- 入口任务、出口任务和孤立任务。
- 任务关系数量及异常关系。

任务级关系用于解释工作流内部结构，不把同一工作流内部存在下游直接视为“跨工作流仍被使用”。

### 5.2 跨工作流依赖

解析 `DEPENDENT` 等依赖任务参数，建立被依赖项目、工作流和任务的反向索引：

- 当前工作流依赖的上游工作流。
- 依赖当前工作流的下游工作流。
- 依赖来源任务、目标代码、解析方式和原始证据摘要。

对不能解析的依赖参数记录失败原因，并将 `dependency_scan_complete` 设为 `false`。

### 5.3 资源和数据依赖补充

从 SQL、Shell 和资源引用中提取可识别的资源路径与表名，作为辅助证据：

- 共享资源文件被其他任务引用。
- 输出表可能被其他工作流读取。
- 解析结果只能提升风险保护，不能单独证明可以下线。

### 5.4 依赖保护规则

- 存在有效跨工作流下游：`protected_by_dependency = true`。
- 依赖扫描不完整：`protected_by_uncertainty = true`。
- 任一保护状态为真时，不得输出“建议下线”，只能输出“保留评估”或“补证确认”。
- 仅在无下游、扫描完整且其他僵尸证据充分时，才允许进入下线确认候选。

## 6. SOP 评分与等级

评分必须保留 `score_total`、`score_version` 和逐项 `score_detail`。第一版建议维度：

| 维度 | 方向 | 说明 |
| --- | --- | --- |
| 长期未更新 | 加疑似分 | 超过 3/6/12 个月阶梯加分 |
| 长期未运行 | 加疑似分 | 最近运行越久，加分越高 |
| 无周期调度或已下线 | 加疑似分 | 表明持续使用可能性较低 |
| 近 30 天零运行 | 加疑似分 | 仅在实例扫描完整时生效 |
| 有有效下游依赖 | 加保护分 | 阻止下线建议 |
| 有资源或数据引用 | 加保护分 | 进入人工确认 |
| 近期仍运行或成功 | 加保护分 | 判定仍活跃 |
| 外部访问证据存在 | 加保护分 | 判定仍被使用 |
| 证据缺失 | 不计零分 | 标记不确定并降级动作 |

等级定义：

- A：高度疑似僵尸，证据完整、无有效下游，可发起下线确认。
- B：长期不活跃，但存在负责人、资源引用、上游关系或其他影响证据，需要重点确认。
- C：证据不足、依赖解析不完整或影响范围不清，进入观察和补证。
- D：近期仍活跃、存在有效下游，或已有明确保留理由，不作为下线候选。

等级和动作分离。`A` 仅代表优先治理，实际动作仍是 `REQUEST_DECOMMISSION_CONFIRMATION`，不是自动下线。

## 7. 治理库模型

建议使用三张逻辑表；若现有治理库已有兼容表，则通过适配层映射，不重复建表。

### 7.1 扫描批次表

记录批次 ID、国家、参数、开始结束时间、扫描数量、候选数量、等级分布、成功状态、错误码和代码版本。

### 7.2 工作流治理明细表

记录工作流身份、负责人、活跃度、调度、运行统计、评分、等级、保护状态、建议动作、治理状态、确认截止时间和历史批次。

### 7.3 依赖证据表

每条依赖单独保存源项目/流程/任务、目标项目/流程/任务、依赖类型、方向、解析状态和证据摘要。详细任务参数不直接进入 n8n 响应。

落库使用单批次事务：候选和依赖成功后再完成批次状态；失败时批次标记失败，不返回虚假成功。

## 8. n8n 接口契约

n8n SSH 节点不再输出完整 `data` 数组，只解析单个摘要对象：

```json
{
  "success": true,
  "batch_id": "20260720-th-001",
  "country": "th",
  "score_version": "v1",
  "scanned_workflows": 12345,
  "candidate_count": 86,
  "persisted_count": 86,
  "level_summary": {"A": 5, "B": 18, "C": 41, "D": 22},
  "dependency_protected_count": 17,
  "uncertain_dependency_count": 3,
  "top_candidates": [],
  "detail_storage": "governance_db"
}
```

`top_candidates` 最多 20 条，只包含工作流身份、等级、总分、负责人、核心原因、上下游计数、保护状态和建议动作。错误响应同样保持小体积，返回错误码、阶段和脱敏消息，不回传 SQL 全文、密码或大段 stdout。

## 9. n8n workflow 改造

- 保留手动触发、参数构建、按国家分流和各国 SSH 凭据。
- 各国 SSH 节点只设置非敏感连接参数并调用统一 Python 脚本。
- 删除 `Parse DS Metadata Result` 的大 JSON 解析逻辑。
- 删除 `Build Zombie Workflow Candidates` 的 JavaScript 聚合逻辑。
- 不再把候选数组传给评分子流程；评分在 Python 内完成。
- 新增统一摘要校验节点，限制响应字段和 `top_candidates` 数量。
- 生成的 workflow JSON 不包含任何明文密码。

## 10. 错误处理和可观测性

- 阶段错误码：连接、结构探测、元数据读取、依赖解析、评分、落库和摘要输出。
- 日志写入跳板机本地滚动文件，stdout 仅输出最终 JSON。
- 日志不得打印密码；任务参数和 SQL 只输出哈希或截断摘要。
- 单个依赖解析失败不终止全批，但必须增加失败计数并触发不确定性保护。
- DS 查询或治理库写入失败时 `success=false`，n8n 不继续通知或下线计划。

## 11. 测试与验收

### 11.1 单元测试

- DS 版本与字段能力识别。
- 任务 DAG 上下游和跨工作流反向索引。
- 依赖参数缺失、损坏和循环依赖。
- 三态外部证据。
- A/B/C/D 评分和依赖保护覆盖。
- 幂等键、摘要字段白名单和 Top N 限制。

### 11.2 集成测试

- 使用固定脱敏夹具模拟大批量工作流，确认内存不会随完整结果无限增长。
- 使用临时治理库验证事务、失败回滚和重跑幂等。
- 验证 n8n workflow JSON 不含密码或大数组传递节点。

### 11.3 上线验收

- 先对单个国家、单个项目执行 `--dry-run`。
- 与现有导出结果抽样核对工作流数、任务数和更新时间。
- 人工核对至少 10 个存在跨工作流依赖的对象和 10 个无依赖候选。
- 确认摘要体积小于 200 KB，且 n8n 不再出现 20 MB 数据解析或 502。
- 开启落库后仍不执行任何 DS 变更。

## 12. 发布与回滚

1. 提交 Python、测试、README 和新版 workflow JSON。
2. 在各国跳板机部署仓库代码并配置环境变量。
3. 单国、单项目 dry-run。
4. 单国全量只读扫描但不落库。
5. 开启治理库落库并检查幂等。
6. 替换 n8n workflow，保留旧 workflow 为停用回滚版本。

回滚只需停用新版 workflow 并恢复旧触发入口；治理明细按批次保留，不删除历史。任何停调度、下线或归档操作均需单独操作计划和用户明确确认。
