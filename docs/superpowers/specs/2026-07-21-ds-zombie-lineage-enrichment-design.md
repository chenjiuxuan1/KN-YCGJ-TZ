# DS 僵尸工作流显式依赖与表血缘增强设计

## 目标

修复 CSV 中“下游数量大于零但显示无下游依赖”的矛盾，并在不改变任何 DS 工作流、调度或任务的前提下，为僵尸候选补充表级下游使用证据。所有增强均为只读。

## 问题与根因

扫描结果中的 `downstream_count`、`review_downstream_count` 和 `dependent_downstream_count` 来源于 Python 扫描器的 DS 显式依赖图；CSV 的 `下游依赖明细` 由 n8n 对扫描器返回字段做兜底。当远端仍运行旧扫描器、未返回详情字段时，n8n 将空字段错误标成“无下游依赖”。

当前扫描也没有表血缘：它不会判断候选工作流写出的表是否被其他活跃 DS 工作流读取。

## 设计

### 第一阶段：显式 DS 依赖一致性

Python 扫描器继续从 `DEPENDENT`、`SUB_PROCESS` 任务参数构建直接跨工作流图，并且每一条关系回填：

- 依赖方向和类型；
- 下游项目、工作流、任务；
- 下游是否活跃及活跃原因；
- 解析状态。

扫描器显式返回 `dependency_detail_status`：

- `available`：详情已完整返回；
- `none`：未发现跨工作流下游；
- `incomplete`：依赖参数解析失败或关系无法定位；
- `unavailable`：扫描器版本未支持详情字段。

n8n 只在状态为 `none` 且总数为零时写“无下游依赖”。若数量大于零但详情缺失，CSV 写“已识别 N 个下游依赖，但明细未返回；请部署最新扫描脚本”，并标记为证据不完整，禁止下线建议。

### 第二阶段：候选写表提取

仅对第一阶段输出的非 D 级候选执行增强。Python 从候选工作流的 SQL、SHELL 任务 `task_params` 中提取：

- 写表：`INSERT INTO`、`INSERT OVERWRITE`、`MERGE INTO`、`CREATE TABLE AS`、`REPLACE INTO`；
- 读表：`FROM`、`JOIN`；
- 资源引用：`resourceList`；
- 标准 SQL、模板 SQL、表集合和解析失败摘要。

注释、反引号、大小写、日期/数字/字符串变量和 CTE 临时名称按现有 SQL 匹配链路规则规范化。无法识别动态 SQL 时只记录不确定性，不推断表血缘。

扫描摘要对每个候选只携带去重后的候选写表及需要补充内容的任务定位信息，不回传完整 SQL。

### 第三阶段：Router 定向表血缘增强

n8n 对每个候选写表按以下顺序调用现有 `ds-scheduler-router`：

1. `search_resource_sql` 搜索包含该表的 DS 资源文件；
2. 资源路径命中而无正文时，调用 `view_resource_file`；
3. 已定位到项目、工作流、任务时，调用 `extract_task_runtime_config`，取得内嵌 SQL 和关联资源；
4. 针对无法由资源中心定位的脚本，复用现有 Git 搜索与 SQL 内容比对能力，国家目录仍为既有六国目录；
5. 仅保留与候选写表匹配、并能关联到活动 DS 工作流的消费者。

这一步不使用异常告警账号的 `e_load` 分流条件：表血缘面向候选工作流的所有写表。资源中心/Git 只是定位脚本的证据源，最终必须尽可能绑定到 DS 项目、工作流、任务。

### 证据分级与治理动作

| 证据 | 条件 | 处理 |
| --- | --- | --- |
| 显式 DS 下游 | `DEPENDENT/SUB_PROCESS` 下游活跃 | D 级硬保护 |
| 表血缘高置信度 | 候选写表被已绑定且活跃的 DS 任务读取 | D 级保护，标记表血缘保护 |
| 表血缘中置信度 | SQL/资源文件确认读表，但工作流绑定或活跃性不完整 | B 或 C，负责人确认/补证据 |
| 表血缘低置信度 | 仅表名文本命中、库名不完整或动态 SQL | C，不能单独保护或下线 |
| 无表血缘证据 | 已完成受限查询但未命中 | 不改变现有评分，仅记录范围与状态 |

任何扫描、Router 或 Git 超时都不被解释为“无下游使用”。

### 限制和性能

- 只增强非 D 级候选，优先 A，再 B，最后 C；
- 每候选最多 10 个写表、每表最多 2 个资源中心命中、最多 2 个任务资源文件；
- 每候选 Router 增强预算 15 秒，单次资源调用 5 秒；
- Git 搜索沿用现有 8 秒预算、文件大小和候选文件上限；
- 达到预算时输出 `lineage_status=partial`，不将未完成查询当成未使用。

### CSV 和 n8n 契约

新增或修订列：

- `显式下游依赖明细（项目/工作流/任务）`；
- `显式依赖明细状态`；
- `候选写入表`；
- `表血缘下游明细（项目/工作流/任务/读取表）`；
- `表血缘置信度`；
- `表血缘证据来源`；
- `表血缘扫描状态`；
- `下游保护依据`。

CSV 不输出原始 SQL、密码、完整任务参数或未脱敏 Git 内容。所有枚举维持中文展示。

## 错误处理

所有阶段使用显式状态而非空字符串：`available`、`none`、`partial`、`incomplete`、`unavailable`、`failed`。依赖明细和表血缘的失败消息需要截断并脱敏。任何失败都只会阻止下线建议或要求补证据，不会触发 DS 修改。

## 测试与验收

- 下游计数大于零但详情缺失时，CSV 不得输出“无下游依赖”；
- 解析出的 `DEPENDENT` 和 `SUB_PROCESS` 必须输出项目、工作流、任务和类型；
- 候选写表与活跃消费者读表匹配时，必须生成高置信度表血缘保护；
- CTE、临时表、跨库同名表、动态 SQL 和超时必须进入受控的不确定状态；
- n8n CSV 的中文枚举、无原始 SQL 和单候选限流契约需要自动测试；
- 菲律宾单项目 dry-run 抽样核对显式依赖和表血缘各 10 条后，才能执行全量只读扫描。

## 发布与回滚

先部署 Python 显式依赖修复，再导入 n8n 增强 workflow；先执行菲律宾单项目 dry-run，再进行全量只读扫描。保留旧 workflow 为停用状态。回滚仅停用增强 workflow 并恢复旧工作流；不会删除治理批次，也不会修改 DS 对象。
